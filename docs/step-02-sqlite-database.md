# Step 2 — SQLite Database with Rich Mock Data

## What this component does
Creates all database tables and fills them with realistic mock business data — orders, revenue, cancellations, city breakdowns, restaurant performance, hourly trends, and manager profiles. This is the data our voice agent will query during briefings.

## Why we designed it this way
- Mock data needs to be **realistic enough to trigger anomalies** — otherwise anomaly detection has nothing to find
- Tables mirror what a real food delivery company tracks
- We include week-on-week comparisons so the agent can say "orders are down 15% vs last week"

## How it connects to the rest
- `data_service.py` (Step 3) queries these tables
- `anomaly_service.py` (Step 8) scans them pre-briefing
- `freshness_service.py` (Step 9) tracks when each metric was last updated
- The 3 sub-agents (Step 11) each query different slices — CEO gets top-line, Ops gets city-level, Analyst gets everything

## Production reality (Swiggy connection)
- Swiggy's Hermes queries Snowflake with charter-based compartmentalization per business unit
- Their tables track P95 delivery times, customer claims by restaurant cohort, region-level metrics
- Our tables mirror this: `orders` by city, `restaurants` with ratings/complaints, `hourly_trends` for temporal patterns
- Swiggy's V2 pipeline breaks queries into: metrics retrieval -> table/column retrieval. Our table design supports this same flow

## Baked-in Anomalies (designed to be detected)

The mock data has 3 anomalies planted so our anomaly detection (Step 8) has something to find:

| Anomaly | What happened | Severity |
|---------|--------------|----------|
| Mumbai orders dropped ~25% yesterday | 12,107 -> 9,000 | High |
| Delhi cancellations spiked ~35% yesterday | avg 420 -> 567 | High |
| Wok Express has 12 complaints in 7 days | Threshold is 8 | Medium |

This is deliberate. In a demo, the agent will say: "I'm seeing a 25% drop in Mumbai orders compared to the day before. Let me dig into that..."

## Key Files

### `database.py` — Connection + Schema

```python
DB_PATH = Path(__file__).parent.parent.parent / "data" / "analytics.db"
```
- Database lives in `backend/data/analytics.db` (gitignored, regenerated by seed)
- Uses `WAL` journal mode — allows concurrent reads while writing (important when the webhook worker writes while the dashboard reads)
- `row_factory = sqlite3.Row` — rows behave like dicts, so you can do `row["city"]` instead of `row[2]`
- All tables use `IF NOT EXISTS` — safe to call `init_db()` multiple times

### `seed_data.py` — Mock Data Generation

- Uses `random.seed(42)` — deterministic. Same data every time you seed. Important for reproducible demos.
- Clears all tables before seeding (idempotent — run it 10 times, same result)
- Anomalies are hardcoded at specific dates/cities so they're always present

### `schemas.py` — Pydantic Models

- Every API request/response has a typed model
- FastAPI uses these for automatic validation and OpenAPI docs
- Example: `WebhookPayload` validates that Bolna sends `call_id` and `event`

## What breaks if we remove it

| If you remove... | What breaks |
|-----------------|-------------|
| `orders` table | Agent has no data to brief on. The entire product is useless. |
| `managers` table | Outbound calls (Step 17) can't find who to call. |
| `calls` table | No call history, no evaluation tracking, no dashboard data. |
| `data_freshness` table | Agent can't tell the user how old the data is. |
| Baked-in anomalies | Anomaly detection demo shows "everything is fine" — boring demo. |
| `random.seed(42)` | Data changes every run. Demo becomes unpredictable. |

## Where we simplified vs production

| Us | Production (Swiggy) |
|---|---|
| SQLite (single file) | Snowflake / Databricks Delta Lake (distributed) |
| 6 cities, 15 restaurants | Thousands of cities, millions of restaurants |
| 14 days of history | Years of history with partitioning |
| Single `seed_data.py` script | ETL pipelines (Airflow, dbt) running hourly |
| `random.seed()` for determinism | Real data from production systems |
| No indexes (tiny data) | Heavily indexed for sub-second queries on billions of rows |

## Interview questions

### "Why SQLite and not PostgreSQL?"
**Answer**: "For a demo/assignment, SQLite eliminates infrastructure setup — no server process, just a file. The table structure is identical to what you'd have in PostgreSQL or Snowflake. Swapping to PostgreSQL would mean changing only `database.py` — no other file touches the connection layer directly."

### "How did you design your mock data?"
**Answer**: "I planted specific anomalies in the data — a 25% order drop in Mumbai, a 35% cancellation spike in Delhi, and a restaurant with excessive complaints. This ensures the anomaly detection system always has something to find during a demo. I used `random.seed(42)` for reproducibility so the demo is predictable every run."

### "How does this relate to Swiggy's data architecture?"
**Answer**: "Swiggy's Hermes uses charter-based compartmentalization — separate metadata per business unit like Swiggy Food vs Instamart. Our tables follow the same pattern — city-level breakdowns, restaurant-level metrics, temporal trends. The key difference is scale: they query Snowflake with billions of rows, we query SQLite with hundreds. But the schema design mirrors theirs."
